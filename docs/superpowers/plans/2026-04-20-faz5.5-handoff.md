## Tamamlandı

### 5.5A — Veri Modeli ✅
- **Airtable şema (manuel, user yaptı):**
  - `Randevular` tablosuna `staffId` (Link to `Staff`, single record, önce Single line text idi dönüştürüldü)
  - `Randevular` tablosuna `staffName` (Lookup → `staffId.name`)
- **[lib/types.ts](lib/types.ts):** `Appointment` interface'ine `staffId?: string` + `staffName?: string` eklendi
- **[lib/airtable.ts](lib/airtable.ts):**
  - `recordToAppointment`: Link array → string (`staffIdArr[0]`), Lookup array → string (`staffNameArr[0]`)
  - `createAppointment`: `staffId` varsa `[staffId]` array olarak yazıyor
  - `updateAppointmentFields`: `staffId` string → `[string]` dönüşüm; `staffName` ignored (Lookup read-only)
- **[lib/staff.ts](lib/staff.ts):** `listActiveStaffForService(service)` + `getStaffById(id)` helper'ları eklendi

### 5.5B — Availability Logic ✅
- **[lib/booking-lock.ts](lib/booking-lock.ts):** `isSlotStillAvailable(..., staffId?)` → verildiyse sadece aynı personelin randevularıyla çakışma
- **[lib/calendar.ts](lib/calendar.ts):**
  - Yeni: `getStaffAwareSlots(date, duration, staffId)` — Airtable tabanlı, GCal bypass
  - `getAvailableSlots(date, duration, staffId?)` — staffId varsa staff-aware, yoksa eski GCal mantığı
  - `findNextAvailableSlots(..., staffId?)` — passthrough
- **Kritik tasarım kararı:** staffId verildiyse **GCal bypass** edilir. Sebep: GCal tek calendar, her randevu oraya yazılıyor → Ayşe'nin 14:00 randevusu Mehmet için 14:00 slotu kapatmaması gerek. Bu yüzden staff-mode'da Airtable source-of-truth.

2. **Mevcut tool'lara `staff_id` opsiyonel parametre ekle:**
   - `check_availability` — description'a: "Müşteri belirli personel istiyorsa staff_id ver"
   - `book_appointment` — staff_id ile rezervasyon
   - `reschedule_appointment` — staff_id değişimi opsiyonel

3. **[app/api/chat/route.ts](app/api/chat/route.ts) executor:**
   - `list_staff_for_service` case: `listActiveStaffForService(service)` çağır, `{ id, name, role }` döndür
   - `check_availability`'e `staff_id` passthrough → `getAvailableSlots(date, duration, staff_id)`
   - `book_appointment`'e staff_id:
     - `isSlotStillAvailable(..., staffId)` ile staff-aware check
     - `createAppointment({ ..., staffId })` ile yaz
     - GCal event summary'ye personel adı ekle (varsa): `${service} - ${customer_name} (${staff_name})`
   - `reschedule_appointment`'e staff_id: `isSlotStillAvailable(..., excludeId, staff_id)` + `updateAppointmentFields({ ..., staffId })`

4. **SYSTEM_PROMPT akış güncelle:**
   ```
   Randevu akışı:
   1. Hizmet öğren
   2. list_staff_for_service çağır
   3. 2+ personel varsa → "Tercih ettiğiniz çalışanımız var mı?" sor + listele + "Fark etmez" seçeneği
   4. 1 personel varsa → otomatik seç, sorma
   5. 0 personel (boş liste) → eski akış (staff_id boş)
   6. "Fark etmez" → staff_id boş
   7. Tarih sor → check_availability(date, service, staff_id?) çağır
   8. Saat onay → book_appointment(..., staff_id?)
   ```

5. **Edge cases:**
   - Pasif personelin listede olmadığı kontrol edilmeli (`listActiveStaffForService` zaten filtriyor — sadece `active && services match`)
   - Bot bir personel önerip müşteri "yok ben X istiyorum" derse → yeni list_staff_for_service değil, mevcut listeden filtre

## Kritik Notlar

### Veri formatı
- Airtable **Link field read** → `string[]` (array of ids). Kod `staffIdArr[0]` alıyor.
- Airtable **Link field write** → `{ staffId: [id] }` veya `{ staffId: [] }` (boşaltmak için)
- Airtable **Lookup field** → salt okunur, yazmaya çalışma (400 döner)

### Geriye uyumluluk
- Mevcut randevularda `staffId` boş → tüm mevcut akışlar eskisi gibi çalışıyor olmalı
- "Fark etmez" = staffId yok = eski davranış
- GCal bypass sadece staffId **verildiyse** devreye giriyor

### Bozmadan çalışması gereken özellikler
- Chat reschedule (1709784 commit) — staff_id eklerken customer_name GCal summary'yi bozma
- Drag-drop (3f44dec commit) — staffId random değişmemeli, mevcut koru
- Booking-lock race condition koruması (d9e8347) — hâlâ çalışıyor olmalı

### Test için örnek Airtable durumu
- En az 2 aktif Staff kaydı olsun (Ayşe + Mehmet gibi)
- Her ikisi de aynı servisleri versin (services array örtüşsün)
- Böylece "tercih var mı?" akışı test edilebilir

### Model önerisi
- Bot akışı (5.5C SYSTEM_PROMPT): Opus 4.7
- UI (5.5D): Sonnet 4.6

### GCal sync — staff değişince?
- PATCH endpoint GCal event summary'yi sadece tarih/saat değişirse yeniler. Eğer sadece `staffId` değişirse (tarih aynı), GCal event summary eski kalır — "Ayşe" yerine eski personel adıyla görünebilir.
- Çözüm (opsiyonel 5.5C veya 5.5E): PATCH'te `staffChanged` de detect et, gerekirse GCal event description güncelle. MVP için atla, post-Faz iş.
